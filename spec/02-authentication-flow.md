# Authentication Flow

## Overview

SignalEdge has two independent auth layers:

1. **User auth** — Google OAuth via NextAuth.js (who can access the dashboard)
2. **Kite auth** — Zerodha OAuth (broker connection for live data)

User auth gates the entire dashboard. Kite auth is required to start strategy runs.

---

## Layer 1: User Authentication (NextAuth + Google OAuth)

### Invite-Only Registration

New users must have an invite code. Codes are generated by admins via `POST /api/invite-codes/generate`.

### Sign-In Flow

```
Browser              Next.js                 Google            FastAPI
  |                    |                        |                  |
  |-- visit / -------->|                        |                  |
  |                    |-- middleware: no session                   |
  |<-- redirect /login |                        |                  |
  |                    |                        |                  |
  |-- click Sign in -->|                        |                  |
  |                    |-- Google OAuth -------->|                  |
  |<-- Google login page                        |                  |
  |-- authenticate --->|<-- callback+token ------|                  |
  |                    |                        |                  |
  |                    |-- POST /api/users/check (X-Internal-Key) ->|
  |                    |<-- { exists: true/false } ----------------|
  |                    |                        |                  |
  | [existing user]    |-- create JWT session   |                  |
  |<-- redirect / -----|                        |                  |
  |                    |                        |                  |
  | [new user]         |-- redirect /invite?email=...              |
  |<-- invite page ----|                        |                  |
  |-- enter code ------>                        |                  |
  |                    |-- POST /api/users/register -------------->|
  |                    |<-- { success: true } -------------------|
  |                    |-- re-initiate Google OAuth               |
  |<-- redirect / -----|                        |                  |
```

### Session Management

- Sessions are JWT cookies managed by NextAuth (`next-auth/jwt`)
- Session strategy: `jwt` (no database adapter)
- Session contents: `email`, `name`, `image`
- All dashboard routes protected by `src/middleware.ts`

### Protected Routes

| Route | Protection |
|-------|------------|
| `/` and all dashboard routes | NextAuth session required |
| `/login` | Public |
| `/invite` | Public (users without a session land here after first Google OAuth) |
| `/api/auth/*` | Public (NextAuth handlers) |
| `/api/proxy/*` | Session verified server-side before forwarding |

## Layer 2: Backend API Security

All Next.js → FastAPI communication uses a shared `BACKEND_INTERNAL_KEY` secret.

### Internal Key Middleware

Every request to `/api/*` on FastAPI requires:
```
X-Internal-Key: <BACKEND_INTERNAL_KEY>
```

The Next.js proxy at `/api/proxy/[...path]` verifies the user session, then adds this header before forwarding. The browser never sends requests directly to the backend.

**Exception**: `/api/signals/stream` uses stream token auth (see below) because `EventSource` cannot send custom headers.

### SSE Stream Token

```
Browser              Next.js              FastAPI
  |                    |                     |
  |-- GET /api/proxy/stream-token ---------->|
  |  (session verified, 60s HMAC token issued)
  |<-- { token, streamUrl } --------------- |
  |                                          |
  |-- EventSource: streamUrl?stream_token=X ->|
  |               (verified inline, no proxy)  |
  |<===== live SSE events ==================|
```

Token format: `base64url(payload).HMAC-SHA256(payload, BACKEND_INTERNAL_KEY)`
Payload: `{"email": "...", "exp": <unix_timestamp>}`
TTL: 60 seconds (only needed for initial handshake)

### Unprotected Backend Routes

`/kite/*` routes are intentionally unprotected — Kite's OAuth servers call `/kite/callback` directly and cannot send the internal key.

## Layer 3: Kite Authentication (Broker Connection)

Required to receive live market data and start strategy runs. Independent of user auth.

### Kite OAuth Flow

```
User                Frontend              Backend               Kite
 |                    |                     |                    |
 |-- Click Connect -->|                     |                    |
 |                    |-- GET /api/auth/kite|                    |
 |                    |  (Next.js route)    |                    |
 |                    |-- GET /kite/login-url                    |
 |                    |<-- { login_url } ---|                    |
 |<-- redirect to Kite login ------------->|                    |
 |-- login on Kite ---|---------------------|----> authenticate ->|
 |                    |                     |<-- callback -------|
 |                    |                     |  ?request_token=X  |
 |                    |                     |-- exchange token ->|
 |                    |                     |<-- access_token ---|
 |                    |                     |-- save to .env     |
 |                    |                     |-- start run        |
 |                    |<-- redirect to / ---|                    |
```

### Token Lifecycle

- Kite access tokens expire daily at **6:00 AM IST**
- `token_expires_at` is computed at run creation time
- Engine detects expiry and stops the run with status `"expired"`
- User must re-authenticate via "Connect Kite" to resume

## Environment Variables

### Frontend (`frontend/.env`)

| Variable | Required | Description |
|----------|----------|-------------|
| `BACKEND_URL` | Yes | FastAPI base URL (server-side only) |
| `BACKEND_INTERNAL_KEY` | Yes | Shared secret with backend |
| `GOOGLE_CLIENT_ID` | Yes | Google OAuth client ID |
| `GOOGLE_CLIENT_SECRET` | Yes | Google OAuth client secret |
| `NEXTAUTH_SECRET` | Yes | JWT signing secret (`openssl rand -base64 32`) |
| `NEXTAUTH_URL` | Optional | Defaults to auto-detected in dev |

### Backend (`backend/.env`)

| Variable | Required | Description |
|----------|----------|-------------|
| `BACKEND_INTERNAL_KEY` | Yes | Must match frontend value |
| `DATABASE_URL` | Yes | PostgreSQL connection string |
| `FRONTEND_BASE_URL` | Yes | For Kite OAuth redirect |
| `KITE_API_KEY` | Yes | Zerodha API key |
| `KITE_API_SECRET` | Yes | Zerodha API secret |
